# 我的个人知识库 Agent 规范

本文件是本仓库唯一的 agent 工作规范。

- `AGENTS.md`：定义知识库的工作流、目录约定和写作规则
- `README.md`：给人看的仓库说明，只讲预览、发布和目录概览
- `scripts/README.md`：给辅助脚本看的说明，不定义知识库规则，也不能覆盖本文件

---

## 目录结构

```text
raw/       原始材料，什么都往里扔，不用分类
wiki/      你维护的知识库，结构由你根据内容决定
log.md     操作日志，只追加
```

`wiki/` 的文件夹结构由你根据内容自然生长，不要预设过多分类。第一次遇到某类内容时，判断最合适的位置，必要时新建文件夹。

当前已经稳定下来的结构如下：

- `wiki/investment/funds/`：基金类项目 / 基金管理人
- `wiki/investment/companies/`：公司类项目
- `wiki/investment/industry-research/`：行业研究
- `wiki/books/`：已手工整理完成的书籍页面

`raw/imports/obsidian-books/` 和 `raw/imports/obsidian-books-ready/` 仍然属于原始材料层，不是知识库页面。

## 三个操作

### INGEST
触发：我说“处理新文件”“开始整理”，或直接告诉你某个文件名。

1. 读取文件内容
2. 先读取 `wiki/index.md` 了解已有结构
3. 判断内容属于什么、涉及哪些实体（公司、人物、话题、概念等）
4. 更新或新建相关 `wiki/` 页面，一份材料通常涉及多个页面
5. 同一实体的信息合并进同一页面，不要产生重复页面
6. 更新 `wiki/index.md`
7. 追加 `log.md`

### QUERY
触发：我问一个具体问题。

1. 先读 `wiki/index.md`
2. 找相关页面并综合回答
3. 只有我明确说“记下来”或“存进 wiki”时，才将答案写回 `wiki/`
4. 其他情况只回答，不修改任何文件

### LINT
触发：我说“检查一下”。

扫描全部 `wiki/` 页面，找出：矛盾信息、可能过期的内容、孤立页面、缺失的交叉引用。输出清单让我决定是否修复。

## 原则

- **综合，不是归档**：同一话题的多份材料要合并提炼，不要逐份存档
- **实体为中心**：围绕公司、人物、话题建页面，而不是围绕文件建页面
- **观点留时间戳**：我的判断会变，记录变化过程比只记结论更有价值
- **结构自然生长**：遇到新类型内容就新建合适的位置，不要强行套现有分类
- **先读 index.md**：每次操作前必须先看已有结构，避免重复
- **聊天不自动入库**：普通对话默认不写回知识库，只有我明确要求才记录
- **“我的观点”只记录用户观点**：保留这个板块，但只有我明确表达并要求记录时才更新；不要把 AI 的综合分析写进去
- **行业研究页不用“我的观点”命名**：行业研究中的阶段性判断、材料抽取和随更新沉淀的认知，统一放在 `认知更新（按时间记录）`，不要再单独开 `材料补充` 一类并列板块；每条都尽量明确附上来源、对象和时间
- **来源信息尽量结构化**：只要原始材料里有明确的会议时间、访谈对象、材料文件名、路演名称等信息，就优先写进页面的 `来源` 或对应记录中，让后续判断都能追溯到具体时间和对象
- **投资相关页面统一用“材料时间”**：`wiki/investment/` 下页面头部的日期统一表示“当前内容所依据的最新材料时间”，不再用入库或整理时间充当 `最后更新`
- **书籍页面统一用“最后阅读”**：`wiki/books/` 下页面头部和 `index.md` 中 Books 区域的日期统一表示最后阅读时间；如果当前材料没有阅读记录，就明确写 `未记录`
- **脚本只辅助 raw，不直写 wiki**：辅助脚本可以同步、筛选、整理 `raw/`，但不能绕过本流程直接批量生成 `wiki/`

## 格式规范

**index.md**

```text
# 知识库索引
更新：YYYY-MM-DD | N 个页面

## [分类名]
- [[页面名]] | 一句话描述 | 更新日期
```

**log.md**

```text
## [YYYY-MM-DD] ingest|query|lint | 简要说明
- 处理了什么，新建/更新了哪些页面
```
